1. Security Architecture Overview

The Tax Practice Application implements a defense-in-depth security model with multiple layers of protection:

1.1 Multi-Layer Security Model

Layer Protection Implementation
Network Layer Transport Security TLS 1.3, HSTS, Certificate Pinning
Application Layer Request Security OWASP Headers, CSP, Rate Limiting
Authentication Layer Identity Verification MFA, Session Management, TPB Verification
Authorization Layer Access Control Row Level Security, RBAC, Client Assignment
Data Layer Data Protection AES-256 Encryption, TFN Protection, Secure Storage
Audit Layer Monitoring Comprehensive Logging, Breach Detection, Compliance Tracking

1.2 Security Principles

  • Zero Trust Architecture: No implicit trust - every request is authenticated and authorized
  • Principle of Least Privilege: Users only access data they explicitly need
  • Defense in Depth: Multiple security layers protect against single point of failure
  • Privacy by Design: Data protection built into every feature from inception
  • Audit Everything: Complete trail of all sensitive data access and modifications

3. Data Protection Layers

3.1 Row Level Security (RLS)

PostgreSQL Row Level Security ensures users can only access authorized data at the database level:

RLS Policy Framework:

Table User Type Access Policy
profiles End User Can only SELECT/UPDATE own profile
profiles Tax Agent Can SELECT assigned client profiles only
questionnaire_responses End User Full CRUD on own responses only
questionnaire_responses Tax Agent Can SELECT/UPDATE assigned client responses
documents End User Full CRUD on own documents only
documents Tax Agent Can SELECT assigned client documents
audit_logs All Users INSERT only (append-only log)
audit_logs Admin SELECT for monitoring purposes

Client-Agent Assignment Control:

  • Explicit Assignment Required: Tax agents must be explicitly assigned to clients
  • No Blanket Access: Being a tax agent doesn't grant access to all client data
  • Assignment Audit: All client-agent assignments are logged with justification
  • Revocable Access: Client assignments can be revoked, immediately removing access
  • Time-Limited Access: Optional expiry dates for temporary agent access

3.2 Encryption Standards

Data at Rest:

  • Database Encryption: AES-256 encryption for entire database (Supabase managed)
  • TFN Encryption: Additional field-level encryption using pgcrypto extension
  • Document Storage: Encrypted blob storage for uploaded documents (Vercel Blob)
  • Backup Encryption: Automated encrypted backups with separate key management

Data in Transit:

  • TLS 1.3: Latest transport layer security for all communications
  • Certificate Pinning: Prevents man-in-the-middle attacks
  • HSTS Enforcement: Strict-Transport-Security header prevents downgrade attacks
  • Secure Cookies: HttpOnly, Secure, SameSite=Strict flags on all cookies

Key Management:

  • Environment Variable Storage: Encryption keys never hardcoded
  • Key Rotation: Automated quarterly key rotation procedures
  • Separate Keys: Different keys for different data types (TFN, documents, sessions)
  • Access Control: Only authorized personnel can access encryption keys

MFA Requirement Policy

Tax Agents: MFA is MANDATORY for all registered tax agents

4.2 Session Management

Secure Session Controls:

  • 2-Hour Timeout: Automatic session expiry for sensitive tax data access

  • Concurrent Session Limit: Maximum 3 active sessions per user
  • Device Fingerprinting: Sessions tied to specific device characteristics
  • Session Revocation: Users can remotely terminate all sessions
  • Activity Extension: Session extends with user activity (up to absolute timeout)

5. Encryption Standards

5.1 TFN (Tax File Number) Protection

Critical Security Measure

Tax File Numbers are the most sensitive data in the system and receive maximum protection through field-level encryption, access logging, and strict authorization requirements.

TFN Security Implementation:

-- TFN Encryption Example
CREATE EXTENSION IF NOT EXISTS pgcrypto;

-- Encrypt TFN before storage
INSERT INTO profiles (tfn_encrypted)
VALUES (pgp_sym_encrypt('123456789', current_setting('app.encryption_key')));

-- Decrypt TFN (requires authorization)
SELECT pgp_sym_decrypt(tfn_encrypted::bytea, current_setting('app.encryption_key'))
FROM profiles
WHERE id = auth.uid()
AND auth.jwt()->>'role' = 'tax_agent';

TFN Access Requirements:

  1. Verified Tax Agent Status: Must have validated TPB registration
  2. Client Assignment: Must be explicitly assigned to the client
  3. MFA Verification: Recent MFA challenge required (within last hour)
  4. Access Justification: Must provide reason for TFN access
  5. Audit Logging: Every TFN access logged with full context

5.2 Document Encryption

Uploaded Document Security:

  • Encrypted Storage: All documents encrypted at rest in Vercel Blob
  • Access Tokens: Time-limited signed URLs for document access (1-hour expiry)
  • Virus Scanning: Automated malware detection on upload
  • File Type Validation: Whitelist of allowed file types (PDF, JPG, PNG only)
  • Size Limits: Maximum 10MB per file, 50MB total per assessment
  • Watermarking: Optional watermarking for sensitive documents

5.3 Database Encryption

Supabase Security Features:

  • Transparent Data Encryption (TDE): Entire database encrypted with AES-256
  • Encrypted Backups: Point-in-time recovery with encrypted snapshots
  • Connection Encryption: SSL/TLS required for all database connections
  • Key Management: Managed by Supabase with hardware security modules (HSMs)

7. Incident Response

7.1 Data Breach Response Workflow

Notifiable Data Breach Timeline

Under the Privacy Act 1988, organizations must notify the OAIC and affected individuals of eligible data breaches as soon as practicable, and no later than 30 days after becoming aware of the breach.

Automated Breach Detection:

  1. Detection (0-1 hours):
    • Real-time monitoring detects unusual access patterns
    • Automated severity assessment (Low/Medium/High/Critical)
    • Immediate notification to security team
    • Incident ticket automatically created
  2. Assessment (1-4 hours):
    • Security team reviews breach details and scope
    • Identification of affected individuals and data types
    • Risk assessment: likelihood of serious harm
    • Decision on whether breach is "eligible data breach" under NDB scheme
  3. Containment (4-8 hours):
    • Isolate affected systems and revoke compromised credentials
    • Prevent further unauthorized access
    • Preserve evidence for investigation
    • Document all containment actions taken
  4. Notification (within 30 days):
    • Notify OAIC if eligible data breach
    • Notify affected individuals via email/secure message
    • Provide recommended steps to protect themselves
    • Update public breach register if required
  5. Remediation (ongoing):
    • Fix security vulnerabilities that allowed breach
    • Enhanced monitoring of affected accounts
    • Offer credit monitoring or identity protection services
    • Update security policies and procedures
  6. Review (30-60 days):
    • Post-incident review and lessons learned
    • Update incident response procedures
    • Staff training on new security measures
    • Report to management and board

7.2 Incident Severity Classification

Severity Criteria Response Time Notification
Critical Mass TFN exposure, database compromise, ransomware Immediate OAIC, all affected individuals, media
High Multiple client records accessed, admin compromise < 1 hour OAIC, affected individuals
Medium Single client record compromise, failed attack attempt < 4 hours Affected individuals, internal review

10.1 Database Security Schema

Key Security Tables:

-- Audit Logging Table
CREATE TABLE audit_logs (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID REFERENCES auth.users(id),
    action TEXT NOT NULL,
    table_name TEXT NOT NULL,
    record_id UUID,
    old_data JSONB,
    new_data JSONB,
    ip_address INET,
    user_agent TEXT,
    justification TEXT,
    timestamp TIMESTAMPTZ DEFAULT NOW()
);

-- Enable RLS on audit_logs (append-only)
ALTER TABLE audit_logs ENABLE ROW LEVEL SECURITY;

CREATE POLICY "Users can insert audit logs"
    ON audit_logs FOR INSERT
    TO authenticated
    WITH CHECK (true);

CREATE POLICY "Admins can view audit logs"
    ON audit_logs FOR SELECT
    TO authenticated
    USING (auth.jwt()->>'role' = 'admin');

-- Client-Agent Assignment Table
CREATE TABLE client_agent_assignments (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    client_id UUID REFERENCES profiles(id),
    agent_id UUID REFERENCES profiles(id),
    assigned_by UUID REFERENCES profiles(id),
    assignment_reason TEXT NOT NULL,
    assigned_at TIMESTAMPTZ DEFAULT NOW(),
    expires_at TIMESTAMPTZ,
    revoked_at TIMESTAMPTZ,
    CONSTRAINT agent_must_be_tax_agent 
        CHECK ((SELECT role FROM profiles WHERE id = agent_id) = 'tax_agent')
);

    user_id UUID PRIMARY KEY REFERENCES auth.users(id),
    mfa_enabled BOOLEAN DEFAULT false,
    mfa_secret TEXT,

10.2 RLS Policy Examples

Comprehensive RLS Implementation:

-- Profiles Table: Users can only see own profile or assigned clients
CREATE POLICY "Users can view own profile"
    ON profiles FOR SELECT
    TO authenticated
    USING (id = auth.uid());

CREATE POLICY "Tax agents can view assigned clients"
    ON profiles FOR SELECT
    TO authenticated
    USING (
        EXISTS (
            SELECT 1 FROM client_agent_assignments caa
            WHERE caa.client_id = profiles.id
            AND caa.agent_id = auth.uid()
            AND caa.revoked_at IS NULL
            AND (caa.expires_at IS NULL OR caa.expires_at > NOW())
        )
    );

-- Questionnaire Responses: Only owner or assigned agent
CREATE POLICY "Users can manage own responses"
    ON questionnaire_responses FOR ALL
    TO authenticated
    USING (user_id = auth.uid())
    WITH CHECK (user_id = auth.uid());

CREATE POLICY "Agents can view assigned client responses"
    ON questionnaire_responses FOR SELECT
    TO authenticated
    USING (
        EXISTS (
            SELECT 1 FROM client_agent_assignments caa
            WHERE caa.client_id = questionnaire_responses.user_id
            AND caa.agent_id = auth.uid()
            AND caa.revoked_at IS NULL
            AND (caa.expires_at IS NULL OR caa.expires_at > NOW())
        )
    );

-- Documents: Strict access control
CREATE POLICY "Users can manage own documents"
    ON documents FOR ALL
    TO authenticated
    USING (user_id = auth.uid())
    WITH CHECK (user_id = auth.uid());

CREATE POLICY "Agents can view assigned client documents"
    ON documents FOR SELECT
    TO authenticated
    USING (
        EXISTS (
            SELECT 1 FROM client_agent_assignments caa
            WHERE caa.client_id = documents.user_id
            AND caa.agent_id = auth.uid()
            AND caa.revoked_at IS NULL
        )
    );

10.3 Middleware Security

Security Headers Implementation:

// middleware.ts - OWASP Security Headers
export async function middleware(request: NextRequest) {
    const response = NextResponse.next();

    // OWASP Recommended Security Headers
    response.headers.set('X-Frame-Options', 'DENY');
    response.headers.set('X-Content-Type-Options', 'nosniff');
    response.headers.set('X-XSS-Protection', '1; mode=block');
    response.headers.set('Referrer-Policy', 'strict-origin-when-cross-origin');
    response.headers.set('Permissions-Policy', 
        'geolocation=(), microphone=(), camera=()');
    
    // Strict Transport Security (HSTS)
    response.headers.set('Strict-Transport-Security', 
        'max-age=63072000; includeSubDomains; preload');
    
    // Content Security Policy
    response.headers.set('Content-Security-Policy', `
        default-src 'self';
        script-src 'self' 'unsafe-inline' 'unsafe-eval';
        style-src 'self' 'unsafe-inline';
        img-src 'self' data: https:;
        font-src 'self' data:;
        connect-src 'self' https://*.supabase.co;
        frame-ancestors 'none';
    `.replace(/\s+/g, ' ').trim());

    // Session timeout check (2 hours)
    const supabase = createMiddlewareClient(request, response);
    const { data: { session } } = await supabase.auth.getSession();
    
    if (session) {
        const sessionAge = Date.now() - new Date(session.created_at).getTime();
        if (sessionAge > 2 * 60 * 60 * 1000) { // 2 hours
            await supabase.auth.signOut();
            return NextResponse.redirect(new URL('/auth/login', request.url));
        }
    }

    return response;
}
